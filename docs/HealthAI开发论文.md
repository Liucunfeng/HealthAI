# 基于视觉大模型的 Android 健康分析 App 开发总结

**作者：刘乐清（大境初级中学）**  
**项目版本：HealthAI 2.2**

---

## 1. 引言 / 项目背景

现在大家越来越关心自己的身材和饮食健康，但普通人往往看不懂自己的身体数据，也不知道一顿饭到底吃了多少热量。比如我身边的人常开玩笑说"又吃胖了"，却说不清到底吃了多少、问题出在哪。找专业营养师或健身教练既贵又不方便，而每个人手里都有一部能拍照的智能手机——手机里其实已经藏着解决这个问题的钥匙。

与此同时，人工智能（特别是能"看懂图片"的大模型）已经很成熟了。我就想：能不能自己动手做一个手机 App，让用户拍一张照片，AI 就能分析出"身材比例怎么样""这顿饭热量高不高、适不适合我"，并给出可执行的建议？与其每天刷短视频看零散的养生知识，不如亲手做一个能天天用的小工具。

这个项目的意义在于：把原本需要专业知识才能得到的健康建议，通过 AI 直接送到普通人的手机上，而且全部在本地完成拍照、选档案、看历史，只在"分析"这一步调用云端 AI。它既是一个真实可用的小工具，也是我学习 AI 应用开发的一次完整实践——从需求到上线，每一步都是我自己踩坑、查资料、改出来的。

---

## 2. 项目目标与功能

我希望这个 App 能实现以下 5 个具体功能：

1. **身材分析**：拍照后，AI 评估体型分类（如苹果型/梨型/匀称型）、身体比例（腿身比、肩腰比等）、体质描述和体脂率估算，并给出改善建议。
2. **食物分析**：拍下饭菜，AI 识别每种食物、估算热量与蛋白质/碳水/脂肪，并结合用户档案判断"适不适合吃"，给出调整建议。
3. **多图分析**：一次最多上传 4 张照片一起分析，AI 综合多张图给出更可靠的结论（比如正面+侧面一起看体型更准）。
4. **多人身体档案**：可保存最多 10 个人的身体数据（身高、体重、年龄、目标等），分析时通过下拉列表选择用谁的档案，家人可以共用一个 App。
5. **历史记录**：自动保存每一次分析结果，以 4 张缩略图的形式展示，点击可展开看详细建议，方便日后对比自己的变化。

这 5 个功能从"能分析"到"好用"，是我逐步加出来的：先保证单张能跑通，再加多图、多人、历史，每一版都解决一个真实痛点。

## 3. 相关知识与技术

这个项目用到的 AI 概念，我用大白话解释一下：

- **大语言模型（LLM）与多模态**：普通的 AI 只能处理文字，而"视觉大模型"能同时理解图片和文字。我用的 Kimi（月之暗面）的 `kimi-k2.6` 模型就属于这一类，可以"看图说话"。它背后其实是先在内部把图片转成模型能理解的信号，再和文字一起推理。
- **提示词（Prompt）**：就是发给 AI 的指令。光说"帮我看看身材"不够，我要求它**只返回一个 JSON 格式的结构化结果**（比如 `{"overall":"...", "bodyType":"..."}`），这样程序才能稳定地把答案拆开、显示到不同位置。
- **API 与 API Key**：AI 服务放在远程服务器上，手机通过"接口（API）"去问它，就像打电话。API Key 相当于账号密码，证明"是我本人在用、费用由我付"。
- **推理（调用接口）**：手机把照片和文字打包发给云端 AI，AI 算完把结果发回来，这步叫"推理"。因为要传图片，所以网络超时时间设得比较长（120 秒）。
- **本地数据库**：分析结果不能每次都重新问 AI（又慢又费钱），所以把历史存在手机里，下次直接看。

用到的工具/平台（不深入原理，只列是什么）：

- **Android + Kotlin**：开发手机 App 的语言和框架；界面用 XML 描述。
- **Retrofit / OkHttp**：在代码里发起网络请求、调用 AI 接口的工具。
- **Room**：把历史记录和身体档案存在手机本地数据库里（基于 SQLite）。
- **Material Design 3**：让界面好看、统一的官方组件库（下拉框、按钮、卡片等）。
- **Kimi 开放平台**：提供视觉大模型 API（需要 API Key，按量计费，新账号有赠送额度）。
- **GitHub Actions**：免费的"云上打包机"，我每次提交代码它自动把 App 编译成安装包（APK），我再下载安装测试。

## 4. 需求分析与方案设计

**用户是谁**：主要是想管理自己健康的人，以及一家人共用（所以需要多人档案）。**典型场景**有两个：一是晚上在家对着镜子自拍，看看最近身材有没有变化；二是每顿饭拍一张，想知道热量超没超标、要不要少吃两口。

**我列了几个具体需求：**
- 拍照要方便，能一次拍多张（正面、侧面一起看更准）；
- 要能区分"这是爸爸的档案还是我的档案"，否则建议会乱套；
- 分析完的历史不能丢，过几天能回头对比。

**整体方案（输入→处理→输出）**：

```
用户输入（照片 + 选中的身体档案）
        ↓
构造提示词（告诉 AI 看什么、按什么格式回答）
        ↓
调用云端视觉大模型（kimi-k2.6）
        ↓
AI 返回 JSON 结果
        ↓
解析并格式化为中文建议 → 显示在屏幕 + 存入本地历史
```

模块划分：① 拍照/选图；② 身体档案选择；③ 调用 AI 分析；④ 解析结果显示；⑤ 历史记录存取。其中"调用 AI"是核心，"解析 JSON"是关键难点——必须保证 AI 的回答能被程序稳定读出来，否则界面会崩溃或显示乱码。为此我让 AI 强制只输出 JSON，并在代码里加了"去掉 markdown 代码块包裹"的容错。

## 5. 实现过程

**步骤 1：搭界面。** 用 Kotlin + XML 做了 5 个页面（身材、食物、我的、历史、设置），底部用导航栏切换。分析页放了一个图片列表（最多 4 张）和一个"分析"按钮。图片框按 3:4 的竖屏比例显示，更符合自拍和饭菜照片的习惯。

**步骤 2：接 AI 接口。** 用 Retrofit 发请求。这里有个关键容错——用户可能把完整的接口地址整个粘进设置框，导致路径重复。我做了自动规整：

```kotlin
// 剥掉末尾的 /chat/completions，否则会和 @POST("chat/completions") 拼成双路径 → 404
if (raw.endsWith("/chat/completions", ignoreCase = true)) {
    raw = raw.removeSuffix("/chat/completions").trimEnd('/')
}
// 漏填 /v1 时自动补上，避免直接 404
val normalized = if (raw.endsWith("/v1", ignoreCase = true)) raw else "$raw/v1"
```

另外 `kimi-k2.6` 对参数很严格：思考模式温度必须 1.0，非思考模式必须 0.6，其它值直接报错 400。所以我对这个模型统一"禁用思考 + 温度 0.6"：

```kotlin
if (model.contains("k2.6", ignoreCase = true)) {
    body.add("thinking", JsonObject().apply { addProperty("type", "disabled") })
    body.addProperty("temperature", 0.6)
} else {
    body.addProperty("temperature", 0.4)
}
```

**步骤 3：把档案（含姓名）一起发给 AI。** 提示词里专门拼了一段用户档案，确保 AI "知道在分析谁"：

```kotlin
fun profileBlock(profile: UserProfile?): String {
    ...
    return """
        用户身体档案：
        - 姓名：${profile.name.ifBlank { "未命名档案" }}
        - 性别：${if (profile.gender == "male") "男" else "女"}
        - 身高：${profile.heightCm} cm
        - 体重：${profile.weightKg} kg
        ...
    """.trimIndent()
}
```

**步骤 4：下拉选择档案。** 用官方 `ExposedDropdownMenu` 样式，点一下就能弹出列表选人：

```xml
<com.google.android.material.textfield.TextInputLayout
    style="@style/Widget.Material3.TextInputLayout.OutlinedBox.ExposedDropdownMenu"
    android:hint="@string/label_select_profile">
    <com.google.android.material.textfield.MaterialAutoCompleteTextView
        android:id="@+id/auto_profile" ... />
</com.google.android.material.textfield.TextInputLayout>
```

**步骤 5：历史存 4 张图。** 给数据库记录新增"图片列表"和"档案名"字段，历史页用 4 个小图展示，不够的用灰色色块占位。这一步还涉及"数据库迁移"：老版本只有 1 张首图，升到新版本时要给旧表加两列，不能把用户以前的历史删掉。

整个项目我迭代了多个版本（1.0 能跑通 → 2.0 多图与多人档案 → 2.1 修图片比例和下拉 → 2.2 把档案名带入分析、历史 4 图），每版都用 Git 提交、用 GitHub Actions 自动打包，再装回手机验证。

**遇到的困难与解决（最真实的探索）：**

1. **一直 404**：原因是 Base URL 被拼成了双路径。解决：上面那样的地址规整。
2. **模型无权限**：默认模型 `moonshot-v1-8k-vision-preview` 没有视觉权限。解决：改用 `kimi-k2.6`。
3. **400 报错**：温度值被模型拒绝。解决：按模型要求设 0.6 并关掉思考。
4. **编译报错**：Kotlin 里写 `$photoDesc` 后面紧跟中文，变量名被"吃"掉了。解决：改成 `${photoDesc}`。
5. **下拉点不开 / 图片比例反了**：下拉漏了官方样式、竖图被写成横图。解决：加样式、把比例改成 3:4（竖:横）。

## 6. 测试与结果

我在真机上做了多轮测试：单张/4 张照片分析、新建和切换不同人的档案、查看历史展开/折叠。比如拍一碗白米饭，AI 能识别出"米饭"并估算约 200 kcal；拍一张自拍，能给出"匀称型/偏瘦型"之类的体型分类和体脂估算；切换不同档案后，给出的建议会随身高体重变化。返回结果都能被程序解析成可读中文，历史记录也能正常保存和回看，说明"调用—解析—存储"这条主链路是通的。

为了确认稳定，我还做了"边界测试"：没选档案就分析（应走通用建议）、只拍 1 张图（历史应有 3 个灰块占位）、填错 API 地址（应提示真实错误而不是笼统的"检查网络"）。这些都能按预期工作。

**客观不足**：① 分析必须联网且有 API Key，且 Kimi 按量收费；② 体脂率只是 AI 估算，不是医学测量；③ 完全依赖云端，没有离线能力；④ 历史图片以 base64 存数据库，记录多了会占空间。

## 7. 总结与展望

通过这个项目，我完整走了一遍"想法 → 设计 → 写代码 → 调接口 → 改 Bug → 迭代版本"的 AI 应用开发流程，真正理解了大模型是怎么被"接"进一个真实产品的，也锻炼了读错误、查文档、做版本管理（Git + CI）的能力。印象最深的是那些报错：404、400、编译失败，每一个都逼着我去读官方文档、看别人怎么填地址和参数，最后才搞明白"AI 不是点一下就灵，而是要按它的规矩来"。

对我来说，最大的收获不是做出了 App，而是学会了"和 AI 协作debug"的思维方式——把模糊的问题拆成可验证的小步，一步步逼近真相。这也让我对计算机和人工智能产生了更真实的兴趣。

未来还可以改进：① 接入本地轻量模型做离线分析，断网也能用；② 增加运动、睡眠等更多健康维度，做成个人健康档案；③ 用图表展示体重/体脂的变化趋势，让历史更有价值；④ 把历史记录导出成图文报告，方便发给教练或医生看。这个项目让我相信：只要愿意动手，初中生也能用 AI 做出真正有用的东西。
